# Phase 04 — Realtime Progress Sync Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Keep the creator-facing progress shell, websocket state, and refresh hydration aligned so live run updates are visible immediately and remain correct after reconnects or reloads.

**Architecture:** Reuse the shared stage/status vocabulary from Plan 01, then tighten the websocket reducer, store updates, and page-level invalidation so the progress shell remains the authoritative explanation of the run. The backend websocket tests stay project-scoped and typed to prevent client-side drift.

**Tech Stack:** React 19, TypeScript, FastAPI websockets, TanStack Query, Vitest, pytest

---

## Context

Phase 4’s progress surface must stay creator-friendly: progress updates should read as waiting-for-review or blocked states, not raw backend stage names, and reconnects must not duplicate or lose state. The current code already has a websocket reducer and a project page invalidation path; this plan tightens that flow so the live shell and refresh path always agree.

Key files: `frontend/app/hooks/useWebSocket.ts`, `frontend/app/stores/editorStore.ts`, `frontend/app/pages/ProjectPage.tsx`, `frontend/app/components/chat/ChatPanel.tsx`, and `backend/tests/test_api/test_websocket.py`.
