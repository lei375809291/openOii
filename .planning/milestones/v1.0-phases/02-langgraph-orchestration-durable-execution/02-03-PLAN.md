# Phase 2 / Plan 03 — Backend recovery and active-run control

> **For agentic workers:** REQUIRED: Use `superpowers:subagent-driven-development` if subagents are available, or `superpowers:executing-plans` to implement this plan.

**Goal:** Expose resumable-run recovery and explicit active-run control on the backend without silent auto-resume.

**Architecture:** Keep the recovery summary server-authored and stage-oriented, then enforce the active-run guard and websocket recovery events on the backend. Preserve the current event names and progress model; the frontend recovery UI is intentionally split into Plan 04 so this plan stays backend-only and easier to validate.

**Tech Stack:** FastAPI, Pydantic, websocket events.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → API | Resume/cancel actions are untrusted until the server validates project ownership. |
| API → websocket payloads | Recovery details must be server-authored, not client-supplied. |
| websocket store → ProjectPage | UI state should reflect server data only, not stale local assumptions. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Spoofing | resume/cancel actions | mitigate | Verify project ownership and active-run state before accepting any continue/cancel request. |
| T-02-08 | Information Disclosure | recovery summary payloads | mitigate | Build recovery summaries server-side and include only stage-oriented data needed for the creator choice. |
| T-02-09 | Tampering | websocket recovery state | mitigate | Validate all outbound event payloads through `WsEvent` and keep the store derived from those payloads only. |
</threat_model>
